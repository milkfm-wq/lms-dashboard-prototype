# Отчёт для наставника: прототип dashboard (v1)

Дата: **30.03.2026**
Проект: **LMS «Наставник»**

## 1) Зачем сделан этот прототип

Собран отдельный интерактивный прототип интерфейса, чтобы:
- быстро согласовать внешний вид и структуру dashboard без блокировки на backend;
- обсудить логику блоков и таблиц на реальном экране, а не абстрактно в тексте;
- зафиксировать комментарии к каждому элементу до интеграции с реальными данными;
- снизить объём переделок после старта online/staging.

Ключевой принцип: **сначала согласуем UX и поведение блоков, потом подключаем данные и API**.

## 2) Что уже подготовлено

### 2.1 Интерфейсный прототип

- `index.html` в папке `reports/dashboard-prototype/`
- Внутри:
  - mock-блоки dashboard и таблиц;
  - поля для комментариев по элементам;
  - общий комментарий по странице целиком;
  - локальное сохранение (`Save Local`);
  - выгрузка результатов обсуждения в:
    - `JSON` (структурированные ответы),
    - `Markdown` (читаемый протокол).

### 2.2 Набор документов для реализации

Подготовлены рабочие спецификации:
- `Dashboard Spec v1.md` — структура и поведение экранов;
- `Data Contract.md` — какие данные ожидает UI (контракты полей);
- `UI-to-DB Mapping.md` — связь UI-элементов с таблицами/полями БД;
- `API Contract.md` — ожидаемые API-эндпоинты и форматы;
- `Role Visibility Matrix.md` — видимость блоков по ролям;
- `Definition of Done - Dashboard.md` — критерии готовности;
- `UX Feedback Log.md` — журнал решений/комментариев.

## 3) Что будет дорабатываться при привязке к данным

При переходе от прототипа к рабочему dashboard будут последовательно выполнены:

1. Уточнение final-спецификации блоков и сценариев по комментариям наставника.  
2. Закрепление окончательного `Data Contract` (какие поля обязательны, какие производные).  
3. Финализация `UI-to-DB Mapping` (таблица/поле/фильтры/агрегации).  
4. Реализация/доработка API под согласованные контракты.  
5. Подключение UI к API и проверка роли/доступов (`mentor/assistant/student`).  
6. Smoke/E2E проверка на staging с реальными тестовыми данными.

## 4) Роль наставника в процессе

От наставника нужен именно продуктовый фидбек:
- какие блоки действительно важны для ежедневной работы;
- какие метрики должны быть “на первом экране”;
- какие фильтры/срезы нужны обязательно;
- какие таблицы/детализация избыточны или наоборот не хватает;
- какие сигналы должны давать “вау-эффект” визуально и по управляемости.

Формат участия:
- открыть прототип;
- заполнить комментарии по блокам;
- нажать:
  1. `Save Local`
  2. `Export JSON`
  3. `Export Markdown`
- передать оба файла команде.

## 5) Доступ к проекту на GitHub (для наставника)

Репозиторий сейчас приватный. Для чтения наставнику нужен доступ как минимум `Read`.

Что сделать в GitHub:
1. Открыть репозиторий `milkfm-wq/lms-nastavnik`.
2. `Settings` → `Collaborators and teams`.
3. `Add people` → ввести GitHub-логин наставника.
4. Выбрать роль:
   - `Read` — только просмотр (рекомендуется для старта),
   - `Write` — если наставник будет сам комментировать/коммитить через PR.
5. Отправить инвайт и дождаться, пока наставник примет приглашение.

Проверка:
- наставник открывает репозиторий по ссылке и видит папку `reports/dashboard-prototype/`.
