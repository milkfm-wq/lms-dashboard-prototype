## 3) Ключевые технические решения

### 3.1 Общий стек
- Frontend/Backend: `Next.js` (единый веб-контур).
- БД и ORM: `PostgreSQL` + `Prisma`.
- Формат API: `REST + JSON`.
- Ролевой доступ: `mentor / assistant / student`.

### 3.2 Подход к данным
- Сразу проектируется многопоточная структура (даже при MVP с одним активным потоком).
- Импортный слой отделён от производной аналитики.
- Базовый ключ сопоставления импортов в MVP: `student_full_name` (в контексте потока).

### 3.3 Подход к релизу
- Разделение pre-production на независимые треки:
  - `D1`: миграция БД на Supabase;
  - `D2`: деплой приложения на Vercel.
- Для dashboard сначала согласуется UX-прототип, затем выполняется полная backend-интеграция.

## 4) Архитектура данных (кратко)

Ядро схемы (фактическая схема в `app/prisma/schema.prisma`):
- пользователи и роли;
- программы/потоки и участники;
- цели и проектные описания;
- импортные сущности (`startup`, `daily`, `tracking`, `session review`);
- операционный слой (`tasks`, `penalties`);
- журналы импортов (`import_runs`);
- аналитический слой (метрики/агрегаты).

Принцип миграций:
- миграции идут пакетами (waves), а не одной большой операцией;
- сначала identity/core, затем прикладные и аналитические слои.

## 5) Импортный контур (кратко)

Основные источники:
- `startup_form`
- `daily_reports`
- `tracking`
- `session_review_source`

Базовый процесс:
1. загрузка файла;
2. валидация строк;
3. сопоставление ученика;
4. upsert/insert в импортный слой;
5. (при необходимости) пост-обработка и пересчёт.

Повторный импорт должен быть безопасным (без дублей по ключевым комбинациям).

## 6) API и доступы (кратко)

API организовано по модулям (`auth`, `programs`, `members`, `imports`, `tasks`, `penalties`, `metrics`, `dashboard`).

Права:
- `mentor`: полный доступ в рамках потока;
- `assistant`: ограниченный операционный доступ;
- `student`: доступ только к собственным данным + ограниченная групповая видимость.

Статус: ролевой контур и read-only ограничения для `assistant/student` реализованы.

## 7) Dashboard: текущая стратегия

Выбран двухэтапный подход:
1. **UX-прототип отдельно от backend** (с комментариями по каждому блоку);
2. итерации с наставником (`JSON + Markdown` фидбек), после чего — интеграция с API/БД.

Что уже готово:
- web-прототип;
- экспорт обратной связи;
- пакет документов (`Dashboard Spec`, `Data Contract`, `UI-to-DB Mapping`, `API Contract`, `Role Matrix`, `DoD`).

## 8) Текущий технический риск

Главный активный риск на сегодня: `D1` (Supabase migration track).

Текущий факт:
- подключение к Supabase подтверждено;
- миграции упираются в доступы/сетевой режим подключения в отдельных сценариях;
- локальный dev-контур возвращён в стабильный режим.

План стабилизации:
- запуск миграций через GitHub Actions (воспроизводимый контур);
- корректная настройка `DATABASE_URL`/`DIRECT_DATABASE_URL` через Secrets;
- после успешного `migrate deploy` — smoke API на staging.
